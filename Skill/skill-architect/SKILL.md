# Skill Architect 2.0
## 极简需求分析器

> 灵感来源：AutoResearch 的 program.md 设计理念
> - 单一模板
> - 固定节奏
> - 直接可执行

---

## 什么时候用

**触发**：用户说"帮我开发一个 Skill"

**排除**：
- 用户只是抱怨问题，没有要做 Skill
- 已有现成 Skill 可满足
- 用户只是想问问题

---

## 4 问框架 (10-15 分钟)

### 第 1 问：WHEN - 场景

```
📐 Skill Architect

问题 1/4：谁会用这个 Skill？他们会说什么触发？

▸ 用户：[谁？如：帮我写代码的人]
▸ 触发：[他们会说什么？如：帮我查天气]
▸ 排除：[什么情况不说？如：问股票天气时不说]
```

**记录**：
```yaml
用户: [目标用户]
触发: [触发语]
排除: [排除条件]
```

---

### 第 2 问：WHAT - 交付

```
✅ 场景确认：[已记录]

问题 2/4：运行完后，用户会看到什么？

▸ 格式：[文本/文件/操作/数据]
▸ 好例：[描述一个好的输出，或粘贴示例]
▸ 坏例：[描述一个差的输出]
```

**记录**：
```yaml
输出:
  格式: [类型]
  好: "[好例子]"
  坏: "[坏例子]"
```

---

### 第 3 问：HOW - 路径

```
✅ 交付确认：[已记录]

问题 3/4：人工做这件完整步骤是什么？

▸ 步骤 1: [做什么]
▸ 步骤 2: [做什么]
▸ 步骤 3: [做什么]
▸ (根据需要增减)

▸ 最可能出错的是？[步骤X] → [处理方式]
```

**记录**：
```yaml
步骤:
  - [步骤1]
  - [步骤2]
  - [步骤3]
风险:
  步骤X: [问题] → [处理]
```

---

### 第 4 问：LIMITS - 边界

```
✅ 路径确认：[X] 步

问题 4/4：什么必须确认？什么绝对不做？

▸ 必须确认：[敏感操作，如删除/支付]
▸ 绝对不做：[禁止操作，如外发数据]
▸ 异常处理：[遇到问题怎么办]
```

**记录**：
```yaml
边界:
  确认: [必须人工确认的]
  禁止: [绝对不做的]
  异常: [异常处理方式]
```

## 使用原则

1. **4 问必完成** - 不可跳过
2. **每次一問** - 不堆砌
3. **好/坏例子** - 比抽象描述更有效
4. **风险必问** - 减少上线踩坑
5. **边界必设** - 安全底线

---

## 示例

**输入**：帮我开发一个天气查询 Skill

**4 问过程**：

1. WHEN：
   - 用户：需要日常天气的人
   - 触发：说"查天气"或发城市名
   - 排除：问股票天气、问历史天气

2. WHAT：
   - 格式：文本
   - 好例：❄️ 北京 3月27日 周五
   天气：晴 8°C~20°C
   建议：适合外出，注意早晚温差
   - 坏例：20度（没有日期没有建议）

3. HOW：
   - 步骤：获取城市 → 调用API → 解析 → 回复
   - 风险：API超时 → 用缓存或提示稍后

4. LIMITS：
   - 确认：用户没给城市时需确认
   - 禁止：绝不保存用户位置
   - 异常：API失败返回"暂时无法查询"

**输出** → 直接生成完整 SKILL.md → 确认开发
