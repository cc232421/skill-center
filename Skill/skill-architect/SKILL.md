# Skill Architect
## 技能开发需求分析器

> **重要**：在开发任何新 Skill 之前，必须先执行此分析器。这是一个过滤器，确保在写代码前业务逻辑已经完全清晰。

---

## 什么时候用 (WHEN)

**触发条件**：
- 用户说"帮我开发一个 Skill"
- 用户说"我想做个自动化"
- 用户说"这个场景需要个技能"

**明确不使用**：
- 用户只是抱怨问题，没有明确要做 Skill
- 用户只是想问问题，不需要自动化
- 已有现成 Skill 可满足需求

---

## 用户视角流程

```
用户：我有个想法
  ↓
触发 Skill Architect
  ↓
回答 7 个核心问题（渐进式）
  ↓
获得完整 SKILL.md 设计稿
  ↓
确认后进入开发阶段
```

---

## 分析流程

### 第 1 问：用户核心诉求

**问题**：用户的核心诉求是什么？

**话术**：
```
📐 Skill Architect - 需求采集

你的 Skill 要解决什么问题？
用户会对这个 Skill 说什么或做什么？

▸ 例：帮我查天气、帮我整理会议纪要、帮我生成 PPT
```

**记录格式**：
```yaml
用户: [谁会用这个 Skill]
诉求: [他们想要什么]
场景: [触发条件]
```

---

### 第 2 问：触发场景与排除条件

**问题**：这个 Skill 在什么情况下触发，什么情况下绝对不触发？

**话术**：
```
✅ 已记录：[用户诉求]

这个 Skill 在什么情况下触发？
什么情况下明确不触发？

▸ 触发例：用户说"查天气"、用户发送位置
▸ 排除例：用户问"股票天气吗"、用户说"不需要"
```

**关键点**：
- "不触发"的边界比"触发"更重要
- 减少误触发的成本远低于处理垃圾输出

---

### 第 3 问：交付定义

**问题**：这个 Skill 完成后，用户会看到什么？

**话术**：
```
✅ 场景确认：[触发条件] 已记录

输出格式（选一个）：
▸ 文本回复 - 一段文字回答
▸ 结构化数据 - JSON/表格/列表
▸ 文件交付 - PPT/报告/代码文件
▸ 操作执行 - 发消息/建文件/调用工具
▸ 混合输出 - 以上组合
```

**记录格式**：
```yaml
输出类型: [文本/文件/操作]
输出格式: [具体格式说明]
```

---

### 第 4 问：成功标准

**问题**：什么样的结果算"好"，什么算"差"？

**话术**：
```
✅ 输出格式：[格式类型]

▸ 好的例子（描述或粘贴）：
  [这里让用户描述一个理想输出]

▸ 坏的例子（描述或粘贴）：
  [这里让用户描述一个差劲输出]

💡 技巧：好/坏对比比抽象定义更有效
```

**关键点**：
- 允许用户粘贴实际例子
- 对比法比抽象标准更容易理解
- 这步是验收标准的锚点

---

### 第 5 问：执行步骤

**问题**：人工做这件事，完整步骤是什么？

**话术**：
```
✅ 成功标准：已记录 [好/坏例子]

一步步描述执行步骤：

步骤 1: [输入]
步骤 2: [处理]
步骤 3: [输出]
... (根据需要)

💡 先写最重要的 3 步，细节可后续补充
```

**记录格式**：
```yaml
执行步骤:
  1. [步骤1]
  2. [步骤2]
  3. [步骤3]
  ...
```

---

### 第 6 问：风险识别

**问题**：最可能出问题的步骤是哪个？什么问题？

**话术**：
```
✅ 执行步骤：已记录 [X] 步

哪个步骤最可能出问题？
- 步骤 [X]: [问题描述]

可能的问题类型：
▸ 数据获取失败
▸ 格式解析错误
▸ 用户输入无效
▸ API 调用超时
▸ 其他：[具体说明]
```

**记录格式**：
```yaml
风险点:
  - 步骤X: [问题] → [处理方式]
```

---

### 第 7 问：边界控制

**问题**：哪些情况必须人工确认？什么绝对不能做？

**话术**：
```
✅ 风险识别：已记录 [风险点]

哪些操作必须人工确认？
▸ [场景1] → 需要确认
▸ [场景2] → 需要确认
▸ [场景3] → 直接执行

什么绝对不能做？
▸ [禁止操作1]
▸ [禁止操作2]

💡 重点：删除/支付/发送等敏感操作必须确认
```

**记录格式**：
```yaml
边界控制:
  require_human:
    - [必须确认的操作]
  never_do:
    - [绝对禁止的操作]
  exceptions:
    - [异常情况处理方式]
```

---

## 使用原则

1. **必须前置** - 任何新 Skill 开发前必须执行
2. **禁止跳步** - 7 个问题必须全部完成
3. **渐进披露** - 每次只问 1 个问题，不堆砌
4. **进度可见** - 每步显示已完成的答案
5. **可退出** - 中途放弃保留进度，可续杯

---

## 质量检查

在进入开发阶段前，确认：
- [ ] 触发条件清晰，无歧义
- [ ] 排除条件完整，无遗漏
- [ ] 输出格式明确，可验证
- [ ] 成功标准有实例支撑
- [ ] 执行步骤完整可执行
- [ ] 风险点已识别并有处理
- [ ] 边界控制无安全漏洞

## 示例

### 输入
用户："帮我开发一个查天气的 Skill"

### 执行过程
1. 诉求：查询指定城市天气
2. 触发：用户说"查天气" + 城市名
3. 输出：文本 + 今日天气 + 建议
4. 好例：北京今天晴，15-22度，适合出行
5. 步骤：获取城市 → 调用API → 解析 → 回复
6. 风险：API超时 → 返回缓存或提示稍后再试
7. 边界：获取位置需确认，删除数据绝对禁止

### 输出
→ 生成完整天气查询 Skill 设计稿 → 确认后开发
